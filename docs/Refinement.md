# Refinement: Маркетплейс онлайн-классов для детей (Outschool RU)

## 1. Матрица граничных случаев (Edge Cases)

| # | Категория | Сценарий | Ожидаемое поведение | Приоритет |
|---|-----------|----------|---------------------|-----------|
| 1 | Поиск | Пустой результат поиска по редкому запросу | Показать блок "Ничего не найдено" + рекомендации похожих классов + предложение создать запрос учителям | Высокий |
| 2 | Бронирование | Два родителя одновременно бронируют последнее место | Оптимистичная блокировка: первый успешный платёж получает место, второму — возврат + предложение waitlist | Критический |
| 3 | Оплата | Таймаут ЮKassa при обработке платежа (>30 сек) | Показать "Платёж обрабатывается", polling статуса каждые 5 сек, через 5 мин — предложить повторить или связаться с поддержкой | Критический |
| 4 | Видео | Учитель не подключается к занятию в течение 10 минут | Автоматическое уведомление учителю (push + SMS), через 15 мин — автоотмена занятия, полный возврат родителям, пометка в профиле учителя | Критический |
| 5 | Видео | Ребёнок отключается от видеозвонка (потеря интернета) | Сохранить сессию 5 мин, показать "Переподключение...", при повторном входе — автоматическое подключение к той же комнате без повторной авторизации | Высокий |
| 6 | Часовые пояса | Родитель из Владивостока (UTC+10) бронирует класс учителя из Калининграда (UTC+2) | Все времена отображаются в локальном часовом поясе пользователя; подтверждение бронирования содержит оба времени; напоминание за 1 час в локальном времени | Высокий |
| 7 | Возврат | Запрос возврата после посещения 2 из 5 занятий курса | Пропорциональный возврат за неиспользованные занятия (3/5 суммы) минус комиссия платформы; если прошло >50% курса — возврат невозможен, только перенос | Высокий |
| 8 | Учитель | Деактивация аккаунта учителя с активными классами | Запретить мгновенную деактивацию; обязать завершить текущие курсы или найти замену; уведомить всех записанных родителей; заморозить выплаты до разрешения | Критический |
| 9 | Регистрация | Родитель пытается зарегистрировать ребёнка младше 3 лет | Валидация возраста на фронте и бэке; показать сообщение "Платформа для детей от 3 лет"; предложить подписку на уведомление о достижении возраста | Средний |
| 10 | Регистрация | Учитель без подтверждённого образования | Разрешить регистрацию, но пометить профиль как "На проверке"; запретить создание классов до верификации документов (макс. 3 рабочих дня) | Высокий |
| 11 | Оплата | Двойное списание из-за повторного нажатия кнопки | Идемпотентный ключ на каждую транзакцию (UUID v4); дедупликация на уровне API; блокировка кнопки после первого клика | Критический |
| 12 | Оплата | Webhook от ЮKassa приходит раньше, чем клиент получил ответ | Обработка webhook обновляет статус в БД; клиентский polling обнаруживает успех; UI обновляется при следующем запросе статуса | Высокий |
| 13 | Видео | Более 30 участников в одном классе (перегрузка Jitsi) | Лимит участников на уровне API (макс. 30 для группового, 1 для индивидуального); при превышении — автоматическое переключение на SFU-режим Jitsi | Высокий |
| 14 | Контент | Учитель загружает материалы >100 МБ | Лимит загрузки 50 МБ на файл, 500 МБ на класс; валидация типа файла (PDF, изображения, видео до 720p); сжатие изображений на лету | Средний |
| 15 | Поиск | Инъекция через строку поиска (XSS/SQL) | Санитизация входа через Zod; Elasticsearch принимает только экранированные запросы; CSP-заголовки блокируют inline-скрипты | Критический |
| 16 | Расписание | Учитель создаёт занятие на время, когда у него уже есть другое | Проверка конфликтов расписания при создании/редактировании; предупреждение с возможностью продолжить (разные платформы) или запрет (та же платформа) | Средний |
| 17 | Возврат | Массовая отмена класса учителем (серия из 10 занятий) | Автоматический полный возврат всем записанным; уведомление каждому родителю; снижение рейтинга учителя; при >3 отменах в месяц — предупреждение/блокировка | Высокий |
| 18 | Аутентификация | Истечение refresh-токена во время активного занятия | Не прерывать видеосессию; обновить токен в фоне; если невозможно — показать модальное окно "Войдите снова" после завершения занятия | Средний |
| 19 | Данные | Родитель запрашивает удаление аккаунта (152-ФЗ) | 30-дневный период "мягкого удаления"; анонимизация персональных данных; сохранение обезличенной статистики; подтверждение по email | Критический |
| 20 | Оплата | Выплата учителю при нулевом балансе платформы | Выплаты через очередь (Bull); при недостатке средств — уведомление финансовому отделу; учителю — статус "Выплата отложена" с датой следующей попытки | Средний |
| 21 | Видео | Запись занятия превышает лимит хранилища | Автоматическое удаление записей старше 30 дней; уведомление учителю за 7 дней; сжатие видео до 720p; лимит 10 ГБ на учителя | Средний |
| 22 | Расписание | Переход на летнее/зимнее время (DST) | Хранение всех времён в UTC; автоматический пересчёт при отображении; уведомление участникам за 1 день до смены времени | Средний |
| 23 | Бронирование | Родитель бронирует класс для ребёнка вне возрастного диапазона | Мягкое предупреждение "Этот класс рассчитан на возраст X-Y, вашему ребёнку Z"; разрешить бронирование с подтверждением; учитель видит пометку | Низкий |

## 2. Стратегия тестирования

### 2.1 Юнит-тесты (целевое покрытие: 80%+)

**Инструменты:** Jest + ts-jest (NestJS), Jest + React Testing Library (Next.js)

| Модуль | Покрытие | Критические пути |
|--------|----------|------------------|
| Аутентификация | 90% | JWT генерация/валидация, refresh-токен ротация, VK/Яндекс OAuth callback |
| Платежи | 95% | Расчёт суммы, комиссия платформы, идемпотентность, webhook-верификация подписи |
| Записи на классы | 85% | Проверка мест, конкурентное бронирование, waitlist логика |
| Поиск | 75% | Формирование Elasticsearch-запросов, фильтрация, ранжирование |
| Расписание | 85% | Конфликты, часовые пояса, повторяющиеся события |
| Уведомления | 70% | Шаблоны, каналы доставки, дедупликация |
| Профили | 80% | Валидация данных, верификация документов |
| Отзывы | 75% | Расчёт рейтинга, модерация контента |

### 2.2 Интеграционные тесты

**Инструменты:** Supertest + Testcontainers (PostgreSQL, Redis, Elasticsearch)

| Тестовый набор | Описание | Зависимости |
|----------------|----------|-------------|
| API Auth Flow | Регистрация → логин → refresh → logout | PostgreSQL, Redis |
| Payment Webhook | Создание платежа → webhook от ЮKassa → обновление статуса | PostgreSQL, mock ЮKassa |
| Search Indexing | Создание класса → индексация → поиск → результат | PostgreSQL, Elasticsearch |
| Enrollment Pipeline | Выбор класса → оплата → запись → уведомление | PostgreSQL, Redis, mock ЮKassa |
| Video Session | Создание комнаты → получение токена → подключение | Mock Jitsi API |
| File Upload | Загрузка материала → валидация → хранение → получение | S3-compatible storage |
| Notification Delivery | Событие → очередь → обработка → отправка | Redis (Bull), mock email/SMS |

### 2.3 E2E-тесты

**Инструменты:** Playwright

| Сценарий | Шаги | Критичность |
|----------|------|-------------|
| Полный путь родителя | Регистрация → поиск → бронирование → оплата → посещение → отзыв | Критический |
| Полный путь учителя | Регистрация → верификация → создание класса → проведение → получение выплаты | Критический |
| Отмена и возврат | Бронирование → отмена → возврат средств → проверка баланса | Высокий |
| Административный | Логин → модерация класса → блокировка пользователя → просмотр статистики | Средний |

### 2.4 Нагрузочные тесты

**Инструменты:** k6

| Тест | Параметры | Критерий успеха |
|------|-----------|-----------------|
| Поиск классов | 10 000 concurrent, 100 RPS | p99 < 500ms, 0% ошибок |
| Бронирование | 1 000 concurrent на один класс | Корректная обработка конкуренции, 0 двойных записей |
| Видеоподключение | 500 одновременных комнат по 10 участников | Подключение < 3 сек, 0% dropped frames |
| API общий | 5 000 RPS mixed endpoints | p99 < 200ms, error rate < 0.1% |
| WebSocket | 10 000 одновременных соединений | Доставка сообщений < 100ms |

### 2.5 Тесты безопасности

| Тест | Инструмент | Частота |
|------|------------|---------|
| Пентест API | OWASP ZAP | Перед каждым релизом |
| Dependency audit | npm audit + Snyk | На каждый PR |
| Secret scanning | gitleaks | На каждый коммит (pre-commit hook) |
| 152-ФЗ аудит | Ручной + чек-лист | Ежеквартально |
| SSL/TLS проверка | testssl.sh | Еженедельно |
| CORS проверка | Автоматический тест | На каждый PR |

## 3. Тестовые сценарии (Gherkin)

### Сценарий 1: Счастливый путь — родитель регистрируется, находит класс, бронирует, посещает и оставляет отзыв

```gherkin
Функция: Полный путь родителя от регистрации до отзыва

  Сценарий: Родитель успешно проходит полный цикл
    Допустим существует опубликованный класс "Рисование для начинающих" с 5 свободными местами
    И класс запланирован на завтра в 15:00 по московскому времени
    Когда я перехожу на страницу регистрации
    И заполняю email "parent@example.com" и пароль "SecureP@ss123"
    И добавляю профиль ребёнка с именем "Маша" и возрастом 8 лет
    И подтверждаю email по ссылке из письма
    Тогда мой аккаунт активирован

    Когда я ввожу "рисование" в строку поиска
    Тогда я вижу класс "Рисование для начинающих" в результатах
    И вижу рейтинг учителя и количество свободных мест

    Когда я нажимаю "Записаться" для ребёнка "Маша"
    И выбираю оплату банковской картой через ЮKassa
    И ввожу данные карты и подтверждаю оплату
    Тогда я вижу подтверждение записи
    И получаю email с деталями занятия
    И количество свободных мест уменьшилось на 1

    Когда наступает время занятия
    И я нажимаю "Присоединиться к занятию"
    Тогда открывается видеокомната Jitsi
    И я вижу учителя и других учеников

    Когда занятие завершено
    И я перехожу на страницу класса
    И ставлю оценку 5 звёзд
    И пишу отзыв "Ребёнку очень понравилось!"
    Тогда отзыв отображается на странице класса
    И средний рейтинг учителя пересчитан
```

### Сценарий 2: Ошибка оплаты в процессе оформления

```gherkin
Функция: Обработка ошибки оплаты

  Сценарий: Платёж отклоняется банком
    Допустим я авторизован как родитель
    И я выбрал класс и нажал "Записаться"
    Когда я ввожу данные карты с недостаточным балансом
    И подтверждаю оплату
    Тогда ЮKassa возвращает статус "canceled"
    И я вижу сообщение "Оплата не прошла. Проверьте баланс карты или попробуйте другой способ оплаты"
    И место в классе НЕ забронировано
    И запись в таблице payments имеет статус "failed"
    И я могу попробовать оплатить снова

  Сценарий: Таймаут при обработке платежа
    Допустим я авторизован как родитель
    И я подтвердил оплату за класс
    Когда ЮKassa не отвечает в течение 30 секунд
    Тогда я вижу сообщение "Платёж обрабатывается, пожалуйста подождите"
    И система проверяет статус платежа каждые 5 секунд
    Когда проходит 5 минут без ответа
    Тогда я вижу "Не удалось подтвердить платёж. Если средства списаны, они будут возвращены автоматически"
    И место в классе освобождается
```

### Сценарий 3: Учитель отменяет класс с записанными учениками

```gherkin
Функция: Отмена класса учителем

  Сценарий: Учитель отменяет занятие за 24+ часа
    Допустим я авторизован как учитель
    И у меня есть класс "Математика 5 класс" с 8 записанными учениками
    И ближайшее занятие через 48 часов
    Когда я нажимаю "Отменить занятие"
    И указываю причину "Болезнь"
    И подтверждаю отмену
    Тогда все 8 родителей получают email и push-уведомление об отмене
    И каждому родителю начисляется полный возврат средств
    И занятие помечается как "Отменено учителем"
    И мой рейтинг надёжности снижается на 0.1 балл

  Сценарий: Учитель отменяет занятие менее чем за 2 часа
    Допустим занятие начнётся через 1 час
    Когда учитель пытается отменить занятие
    Тогда система показывает предупреждение о штрафных санкциях
    И при подтверждении — полный возврат родителям
    И штраф учителю 500 рублей
    И рейтинг надёжности снижается на 0.5 балл
```

### Сценарий 4: Попытка несанкционированного доступа

```gherkin
Функция: Защита от несанкционированного доступа

  Сценарий: Попытка доступа к чужому профилю ребёнка
    Допустим я авторизован как родитель с id=100
    Когда я отправляю GET-запрос на /api/children/999
    И ребёнок с id=999 принадлежит родителю с id=200
    Тогда я получаю HTTP 403 Forbidden
    И в логах фиксируется попытка несанкционированного доступа
    И мой IP добавляется в мониторинг подозрительной активности

  Сценарий: SQL-инъекция через строку поиска
    Когда я отправляю поисковый запрос "'; DROP TABLE users; --"
    Тогда запрос санитизируется через Zod-валидацию
    И Elasticsearch получает экранированную строку
    И возвращается пустой результат поиска
    И инцидент логируется в систему мониторинга

  Сценарий: Brute-force атака на логин
    Когда с одного IP поступает 10 неудачных попыток логина за 1 минуту
    Тогда IP блокируется на 15 минут
    И владелец аккаунта получает email о подозрительной активности
    И инцидент логируется в SIEM
```

### Сценарий 5: Поиск под высокой нагрузкой

```gherkin

    И 10000 пользователей одновременно выполняют поиск

    И ни один запрос не возвращает ошибку 5xx

```

### Сценарий 6: Конкурентное бронирование последнего места


### Сценарий 7: Возврат средств после частичного прохождения курса

```gherkin
Функция: Пропорциональный возврат

  Сценарий: Возврат после 2 из 5 занятий
    Допустим я оплатил курс из 5 занятий за 5000 рублей
    И посетил 2 занятия
    Когда я запрашиваю возврат средств
    Тогда система рассчитывает возврат: (5000 / 5) * 3 = 3000 рублей
    И удерживает комиссию платформы 5%: 3000 * 0.95 = 2850 рублей
    И возврат 2850 рублей отправляется через ЮKassa
    И я получаю подтверждение возврата на email
```

## 4. Оптимизация производительности

### 4.1 База данных (PostgreSQL)

#### Стратегия индексирования

```sql
-- Поиск классов по категории и статусу (самый частый запрос)
CREATE INDEX idx_classes_category_status ON classes (category_id, status)
  WHERE status = 'published';

-- Поиск по возрастному диапазону
CREATE INDEX idx_classes_age_range ON classes
  USING GIST (int4range(min_age, max_age));

-- Расписание учителя (проверка конфликтов)
CREATE INDEX idx_schedules_teacher_time ON schedules (teacher_id, start_time, end_time);

-- Записи на занятия (проверка мест)
CREATE INDEX idx_enrollments_class_status ON enrollments (class_id, status)
  WHERE status = 'active';

-- Поиск платежей по статусу (для webhook-обработки)
CREATE INDEX idx_payments_idempotency ON payments (idempotency_key)
  WHERE status = 'pending';

-- Полнотекстовый поиск (fallback если Elasticsearch недоступен)
CREATE INDEX idx_classes_search ON classes
  USING GIN (to_tsvector('russian', title || ' ' || description));
```

#### Оптимизация запросов

| Проблема | Решение |
|----------|---------|
| N+1 запросы при загрузке классов с учителями | Prisma `include` с `select` для ограничения полей |
| Медленная пагинация (OFFSET) | Cursor-based пагинация по `id` или `created_at` |
| Тяжёлые аналитические запросы | Материализованные представления, обновление каждые 5 мин |
| Lock contention при бронировании | `SELECT ... FOR UPDATE SKIP LOCKED` + advisory locks |

#### Connection Pooling

```
# PgBouncer конфигурация
pool_mode = transaction
default_pool_size = 25
max_client_conn = 200
reserve_pool_size = 5
reserve_pool_timeout = 3
```

### 4.2 Кэширование (Redis)

| Слой | Ключ | TTL | Инвалидация |
|------|------|-----|-------------|
| Сессии | `session:{userId}` | 24 часа | При логауте / смене пароля |
| Каталог классов | `classes:list:{filters_hash}` | 5 мин | При создании/обновлении класса |
| Профиль учителя | `teacher:{id}:profile` | 15 мин | При обновлении профиля |
| Результаты поиска | `search:{query_hash}` | 2 мин | При изменении индекса |
| Количество мест | `class:{id}:seats` | Без TTL | Атомарный DECR/INCR при бронировании |
| Rate limiter | `ratelimit:{ip}:{endpoint}` | 1 мин | Автоматически по TTL |
| Расписание на сегодня | `schedule:today:{userId}` | 1 час | При изменении расписания |

**Стратегия инвалидации:** Cache-aside pattern. Запись в БД → удаление ключа кэша → следующий read-запрос наполняет кэш.

### 4.3 Фронтенд (Next.js)

| Оптимизация | Реализация |
|-------------|------------|
| Code Splitting | Динамические импорты для страниц учителя, админки, видео |

### 4.4 Видео (Jitsi)

| Оптимизация | Описание |
|-------------|----------|
| Adaptive Bitrate | Simulcast: 3 слоя (180p/360p/720p), автопереключение по bandwidth |
| Fallback | При bandwidth < 500kbps — только аудио + слайды |
| Lazy Join | Видеопоток запускается по нажатию "Присоединиться", не при загрузке страницы |

## 5. Укрепление безопасности

### 5.1 Валидация входных данных

```typescript
// Пример Zod-схемы для создания класса
const CreateClassSchema = z.object({
  title: z.string().min(5).max(200).trim(),

  schedule: z.array(ScheduleSlotSchema).min(1).max(52),

  message: "Максимальный возраст должен быть >= минимального",
});
```

### 5.2 Rate Limiting

| Endpoint | Лимит | Окно |
|----------|-------|------|
| POST /auth/login | 5 запросов | 1 минута |

| * (по умолчанию) | 100 запросов | 1 минута |

**Реализация:** NestJS `@nestjs/throttler` + Redis store для распределённого rate limiting.

### 5.3 Защита от атак

| Вектор | Защита |
|--------|--------|
| XSS | CSP-заголовки, DOMPurify для пользовательского контента, HttpOnly cookies |
| CSRF | CSRF-токены (csurf) для мутирующих запросов, SameSite=Strict cookies |
| SQL Injection | Prisma ORM (параметризованные запросы), Zod-валидация |
| File Upload | Проверка MIME-type + magic bytes, лимит размера, антивирус ClamAV |
| Clickjacking | X-Frame-Options: DENY, frame-ancestors 'none' |
| SSRF | Whitelist внешних URL, запрет обращений к внутренним IP |
| Brute Force | Rate limiting + CAPTCHA после 3-й неудачной попытки |

### 5.4 Соответствие 152-ФЗ

| Требование | Реализация |
|------------|------------|
| Согласие на обработку ПД | Чекбокс при регистрации, версионирование политики |
| Хранение на территории РФ | VPS в российском дата-центре (AdminVPS/HOSTKEY) |
| Право на удаление | API endpoint DELETE /api/me с 30-дневным grace period |
| Шифрование ПД | AES-256 для чувствительных полей в БД |
| Журнал доступа | Audit log для всех операций с ПД |

## 6. Элементы технического долга

| # | Элемент | Приоритет | Описание | План устранения |
|---|---------|-----------|----------|-----------------|
| 1 | Монолитный API | Средний | Все модули в одном NestJS-приложении; при росте до 100+ эндпоинтов — время сборки и тестирования увеличивается | Подготовить разделение на микросервисы (payments, notifications) после MVP |
| 2 | Синхронные webhook | Высокий | Webhook от ЮKassa обрабатывается синхронно в HTTP-хендлере | Перевести на Bull-очередь с retry и dead-letter |
| 3 | Отсутствие API-версионирования | Средний | Текущий API без версий (/api/classes вместо /api/v1/classes) | Добавить версионирование до появления мобильных клиентов |
| 4 | Жёсткая привязка к Jitsi | Низкий | Видеомодуль напрямую использует Jitsi API без абстракции | Создать VideoProvider-интерфейс для возможной замены |
| 5 | Отсутствие feature flags | Средний | Нет механизма постепенного раскатывания фич | Интегрировать Unleash (self-hosted) или простой JSON-конфиг |
| 6 | Ручной деплой | Высокий | Docker Compose через SSH без автоматизации | Настроить GitHub Actions с автодеплоем на staging |
| 7 | Единая БД | Низкий | Все данные в одной PostgreSQL; при >1M пользователей — bottleneck | Подготовить шардирование или read-replicas на этапе v2.0 |
| 8 | Отсутствие CDN | Средний | Статические ресурсы раздаются с VPS | Подключить CDN (Selectel CDN или Cloudflare) для статики и изображений |
| 9 | Email через SMTP | Низкий | Прямая отправка через SMTP без очереди | Перевести на очередь + retry + bounce-обработка |
| 10 | Тесты без seed-данных | Средний | Каждый тест создаёт свои данные, дублирование фикстур | Создать единый seed-файл и фабрики (factory pattern) |
