# Уточнение: Интеграция платежей

## 1. Граничные случаи (Edge Cases)

### EC-1: Идемпотентность вебхуков
**Сценарий:** ЮKassa отправляет один и тот же webhook 3 раза из-за таймаута.
**Решение:** Уникальный индекс на `WebhookLog.eventId`. Первый запрос обрабатывается, последующие — 200 OK без действий. Проверка в начале транзакции.

### EC-2: Таймаут оплаты
**Сценарий:** Родитель открыл страницу ЮKassa, но не завершил оплату в течение 30 минут.
**Решение:** Payment остается в status=pending. Cron-задача через 1 час проверяет статус в ЮKassa API. Если оплата не завершена — Payment.status=expired, enrollment удаляется.

### EC-3: Частичный возврат
**Сценарий:** Ребенок посетил 3 из 10 занятий, родитель запрашивает возврат.
**Решение:** Администратор рассчитывает сумму возврата пропорционально непосещенным занятиям. Partial refund через ЮKassa API. TeacherBalance корректируется на пропорциональную долю.

### EC-4: Точность Decimal-арифметики
**Сценарий:** Цена 999.99 руб., комиссия 22.5%. Результат: 224.99775 → округление.
**Решение:** Все расчеты через `decimal.js` с `toDecimalPlaces(2, ROUND_HALF_UP)`. Prisma Decimal column. Frontend отображает через `Intl.NumberFormat('ru-RU')`.

### EC-5: Конкурентные платежи за одну секцию
**Сценарий:** Два родителя одновременно оплачивают последнее место.
**Решение:** Checkout создает enrollment с status=pending_payment. Место резервируется на 30 минут. Второй родитель видит "место зарезервировано". Если первый не оплатил — место освобождается.

### EC-6: Выплата при недостаточном балансе платформы
**Сценарий:** Учитель запрашивает выплату, но средства еще не поступили от ЮKassa.
**Решение:** Available balance считается только по completed payments. Pending payments не учитываются.

## 2. Пути ошибок

| Ошибка | HTTP | Действие |
|--------|------|----------|
| ЮKassa API недоступен | 502 | Retry через 5s, показать "Попробуйте позже" |
| Невалидная подпись вебхука | 401 | Логировать инцидент, не обрабатывать |
| Payment не найден по yukassaId | 404 | Логировать ALERT, ручное расследование |
| Баланс < минимальной выплаты | 422 | Показать минимальную сумму |
| BullMQ payout failed 3x | — | Dead letter queue + алерт администратору |

## 3. Безопасность

| Угроза | Защита |
|--------|--------|
| Подделка вебхука | HMAC-SHA256 верификация подписи |
| Подмена суммы на клиенте | Сумма рассчитывается только на сервере |
| SQL injection через metadata | Параметризованные запросы (Prisma) |
| Утечка YUKASSA_SECRET_KEY | Хранение в env, не в коде и не в git |
| Replay attack на webhook | Idempotency по eventId |
| Несанкционированный возврат | Только роль Admin через RBAC |

## 4. Аудит и соответствие

- Все финансовые операции логируются с timestamp, userId, amount, status
- Чеки 54-ФЗ формируются через ЮKassa receipt API
- Retention: финансовые логи хранятся минимум 5 лет
- Ежемесячная сверка: сумма payments === сумма teacherEarnings + platformFees

## 5. Производительность

- Webhook endpoint: ответ < 5 секунд (требование ЮKassa)
- Индекс на `Payment.yukassaId` для быстрого поиска по вебхуку
- Индекс на `Payment.parentId` + `Payment.status` для страницы истории
- BullMQ concurrency: 5 параллельных выплат
