# Спецификация: Интеграция платежей

**Feature ID:** payment-integration
**User Stories:** US-P06, US-T05
**Epic:** E4
**Приоритет:** Критический (монетизация)

---

## 1. Функциональные требования

### FR-1: Оплата курса (checkout)
Родитель оплачивает курс через ЮKassa. Создается платеж, родитель перенаправляется на страницу оплаты ЮKassa. При успехе — enrollment подтверждается.

### FR-2: Обработка вебхуков ЮKassa
Система принимает вебхуки `payment.succeeded` и `refund.succeeded`. Подпись верифицируется через HMAC-SHA256. Обработка идемпотентна.

### FR-3: Возврат платежа (refund)
Администратор инициирует полный или частичный возврат. Запрос отправляется в ЮKassa, enrollment переходит в соответствующий статус.

### FR-4: Доходы учителя (earnings)
Учитель видит детализацию доходов по периодам. Комиссия платформы вычитается автоматически.

### FR-5: Выплата учителю (withdrawal)
Учитель запрашивает выплату накопленных средств. Заявка обрабатывается через BullMQ очередь с retry-логикой.

## 2. Acceptance Criteria (Gherkin)

```gherkin
Feature: Интеграция платежей

  Scenario: Успешная оплата курса
    Given родитель выбрал секцию и ребенка
    When родитель нажимает "Оплатить" и перенаправляется на ЮKassa
    And оплата проходит успешно
    Then ЮKassa отправляет webhook payment.succeeded
    And система создает Payment со status=completed
    And enrollment переходит в status=confirmed
    And учителю начисляется сумма за вычетом комиссии платформы
    And родитель получает чек на email

  Scenario: Неуспешная оплата
    Given родитель перенаправлен на ЮKassa
    When оплата отклонена
    Then enrollment остается в status=pending
    And родитель видит ошибку с возможностью повторить

  Scenario: Возврат платежа
    Given администратор инициирует возврат для Payment
    When запрос refund отправлен в ЮKassa и подтвержден
    Then Payment.status переходит в refunded
    And сумма вычитается из баланса учителя
    And родитель получает уведомление о возврате

  Scenario: Идемпотентность вебхуков
    Given webhook с idempotencyKey уже обработан
    When ЮKassa повторно отправляет тот же webhook
    Then система возвращает 200 OK без повторной обработки

  Scenario: Запрос выплаты учителем
    Given учитель имеет баланс >= 1000 руб.
    When учитель нажимает "Запросить выплату"
    Then создается задача в BullMQ очереди payouts
    And статус выплаты отображается как "В обработке"
```

## 3. Нефункциональные требования

| NFR | Требование |
|-----|-----------|
| Точность | Все финансовые расчеты — Decimal (decimal.js), НИКОГДА float |
| Безопасность | HMAC-SHA256 верификация вебхуков, secrets в env |
| Надежность | BullMQ: retry 3 раза с exponential backoff, dead letter queue |
| Аудит | Все финансовые операции логируются с полным контекстом |
| Соответствие | Чеки по 54-ФЗ через ЮKassa |

## 4. Комиссионная модель

| Параметр | Значение |
|----------|----------|
| Комиссия платформы (учитель) | 20-25% от цены занятия |
| Marketplace fee (родитель) | 5-10% |
| Минимальная выплата | 1000 руб. |
| Период выплат | По запросу, обработка до 3 рабочих дней |
