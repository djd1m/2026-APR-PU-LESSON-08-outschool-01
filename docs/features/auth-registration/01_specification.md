# Спецификация: Аутентификация и регистрация

## Пользовательские истории

### US-P01: Регистрация и вход родителя

**Как** родитель,
**я хочу** зарегистрироваться по email/паролю или через VK/Яндекс,
**чтобы** получить доступ к каталогу и записи ребенка на занятия.

**Приоритет:** Critical | **Effort:** L

### US-T01: Регистрация и вход учителя

**Как** учитель,
**я хочу** зарегистрироваться с указанием роли TEACHER,
**чтобы** получить доступ к личному кабинету и создавать занятия.

**Приоритет:** Critical | **Effort:** M

## Критерии приемки (Gherkin)

```gherkin
Feature: Регистрация по email/паролю

  Scenario: Успешная регистрация нового пользователя
    Given пользователь на странице регистрации
    When он вводит email "parent@example.com" и пароль "SecurePass1!"
    And выбирает роль "PARENT"
    And нажимает "Зарегистрироваться"
    Then система создает аккаунт с хешированным паролем (bcrypt)
    And возвращает access-токен (JWT RS256, TTL 15 мин) и refresh-токен (TTL 7 дней)
    And устанавливает httpOnly secure cookie с access-токеном

  Scenario: Регистрация с уже существующим email
    Given в базе существует пользователь с email "existing@example.com"
    When новый пользователь пытается зарегистрироваться с тем же email
    Then система возвращает 409 Conflict с сообщением "Email уже зарегистрирован"

  Scenario: Вход по email/паролю
    Given зарегистрированный пользователь с email "parent@example.com"
    When он вводит корректные email и пароль
    Then система возвращает новую пару access + refresh токенов

  Scenario: Неверный пароль при входе
    Given зарегистрированный пользователь
    When он вводит неверный пароль
    Then система возвращает 401 Unauthorized
    And увеличивает счетчик неудачных попыток

Feature: OAuth-вход через VK ID и Яндекс ID

  Scenario: Первый вход через VK ID
    Given пользователь нажимает кнопку "Войти через VK"
    When VK возвращает callback с profile данными
    And пользователя с providerId нет в базе
    Then система создает новый аккаунт (provider=VK, providerId=...)
    And возвращает JWT-токены

  Scenario: Повторный вход через OAuth
    Given пользователь уже входил через Яндекс ID
    When Яндекс возвращает callback с тем же providerId
    Then система находит существующий аккаунт
    And возвращает JWT-токены без создания дубликата

Feature: Обновление токенов

  Scenario: Обновление access-токена по refresh-токену
    Given у пользователя истек access-токен
    When он отправляет POST /auth/refresh с валидным refresh-токеном
    Then система возвращает новый access-токен
    And ротирует refresh-токен (старый инвалидируется)

  Scenario: Невалидный refresh-токен
    Given refresh-токен отозван или просрочен
    When пользователь пытается обновить access-токен
    Then система возвращает 401 и требует повторный вход
```

## Нефункциональные требования

- Пароль: минимум 8 символов, 1 заглавная, 1 цифра, 1 спецсимвол
- bcrypt cost factor: 12
- JWT RS256 с ротацией ключей раз в 90 дней
- Rate limiting: 5 попыток входа в минуту на IP
- Все эндпоинты через HTTPS
