## Детальный анализ проблемных историй

### US-P04: Просмотр карточки класса --- 65/100 (FAIR)

**INVEST-анализ:**

| Критерий | Результат | Комментарий |
|----------|:---------:|-------------|
| Independent | PASS | Может разрабатываться отдельно |
| Negotiable | PASS | Состав полей обсуждаем |
| Valuable | PASS | Критически важно для конверсии |
| Estimable | FAIL | Нет чётких NFR-метрик (время загрузки, размер страницы) |
| Small | PASS | Укладывается в спринт |
| Testable | PASS | Есть список обязательных полей |

**SMART-анализ:**

| Критерий | Результат | Комментарий |
|----------|:---------:|-------------|
| Specific | PASS | Перечислены конкретные элементы |
| Measurable | FAIL | Нет метрик загрузки страницы и отклика |
| Achievable | PASS | Стандартная карточка товара |
| Relevant | PASS | Ключевая для конверсии |
| Time-bound | FAIL | Нет временных ограничений на загрузку |

**Рекомендации:**
1. Добавить метрику: "Карточка класса загружается за < 2 сек (P95)"
2. Добавить метрику: "SEO-оптимизированная мета-информация (title, description, OG-tags)"
3. Указать поведение при отсутствии данных (нет отзывов, нет фото)
4. Добавить критерий доступности (WCAG 2.1 AA --- Alt-тексты для изображений)

### US-P07: Посещение видеозанятия --- 65/100 (FAIR)

**INVEST-анализ:**

| Критерий | Результат | Комментарий |
|----------|:---------:|-------------|
| Independent | FAIL | Сильно зависит от US-P06 (оплата) и инфраструктуры Jitsi |
| Negotiable | PASS | Технические детали обсуждаемы |
| Valuable | PASS | Ключевая функция платформы |
| Estimable | PASS | Есть чёткие технические параметры |
| Small | FAIL | Интеграция Jitsi --- сложная задача, может не уложиться в 1 спринт |
| Testable | PASS | Есть метрики (задержка < 200ms) |

**SMART-анализ:**

| Критерий | Результат | Комментарий |
|----------|:---------:|-------------|
| Specific | PASS | Описаны функции видеокомнаты |
| Measurable | PASS | Задержка < 200ms |
| Achievable | PASS | Jitsi / LiveKit --- зрелые решения |
| Relevant | PASS | Основная функция платформы |
| Time-bound | FAIL | Нет времени на создание комнаты, нет SLA на uptime |

**Безопасность (-10):** История работает с детьми в видеосреде, но отсутствуют:
- Ограничение доступа: вход только по ссылке + PIN (описано в PRD F4, но отсутствует в AC истории)
- Автоматическое завершение комнаты
- Родительское уведомление при подключении ребёнка
- Запрет прямых сообщений между детьми и преподавателем вне контролируемой среды

**Рекомендации:**
1. Добавить AC: "Вход в комнату возможен только для пользователей с активным enrollment"
2. Добавить AC: "Комната автоматически закрывается через 10 мин после окончания занятия"
3. Добавить AC: "Родитель получает push-уведомление при входе ребёнка в комнату"
4. Разбить историю: отдельно "создание комнаты" и "участие в занятии"
5. Добавить fallback-сценарий при сбое видео

### US-C01: Присоединение к классу и геймификация --- 62/100 (FAIR)

**INVEST-анализ:**

| Критерий | Результат | Комментарий |
|----------|:---------:|-------------|
| Independent | FAIL | Объединяет два разных домена: участие в занятии и геймификацию |
| Negotiable | PASS | Правила начисления XP обсуждаемы |
| Valuable | PASS | Мотивация детей --- ключевой дифференциатор |
| Estimable | PASS | Формулы XP конкретны |
| Small | FAIL | Два домена в одной истории --- лучше разбить |
| Testable | PASS | Конкретные значения XP и уровней |

**SMART-анализ:**

| Критерий | Результат | Комментарий |
|----------|:---------:|-------------|
| Specific | PASS | Конкретные значения XP |
| Measurable | PASS | 10 XP за посещение, 100 XP на уровень |
| Achievable | PASS | Стандартная геймификация |
| Relevant | PASS | Увеличивает retention |
| Time-bound | FAIL | Нет временных рамок начисления (real-time vs batch?) |

**Безопасность (-10):** Ребёнок является основным пользователем, но отсутствуют:
- Возрастные ограничения на интерфейс (436-ФЗ)
- Ограничение по авторизации: ребёнок входит только через аккаунт родителя
- Контроль чрезмерного использования (экранное время)

**Рекомендации:**
1. Разбить на 2 истории: "US-C01a: Участие в занятии" и "US-C01b: Начисление XP и бейджей"
2. Добавить AC: "Ребёнок авторизуется исключительно через аккаунт родителя (436-ФЗ)"
3. Добавить AC: "XP начисляется в течение 5 секунд после завершения занятия"
4. Добавить AC: "Родитель может ограничить экранное время ребёнка в настройках"
5. Уточнить, что считается "активным участием" (просто подключение или взаимодействие?)

## Рекомендации

### Приоритет 1 (Блокеры --- исправить до реализации)

1. **Устранить расхождение PRD/Specification по Яндекс ID:** определить, входит ли Яндекс ID в MVP или v1.0. PRD указывает P0 (MVP), Feature Matrix --- v1.0. Привести в соответствие.

2. **Устранить расхождение по геймификации:** PRD указывает P1 (Should Have для MVP), Feature Matrix сдвигает XP/бейджи/streak на v1.0. Согласовать с продакт-менеджером.

3. **Добавить критерии безопасности в US-P07 (видеозанятия):** вход только для оплативших, автоматическое завершение комнаты, уведомление родителю.

4. **Добавить критерии безопасности в US-C01 (геймификация для детей):** авторизация через родителя, соответствие 436-ФЗ.

### Приоритет 2 (Значительные --- исправить в текущей итерации)

5. **Унифицировать SLA поиска:** PRD требует < 200ms, Specification --- < 500ms. Рекомендуем < 500ms для полнотекстового, < 200ms для фильтров.

6. **Добавить отдельную US для AI-рекомендаций (F8):** функция AI-matching описана в PRD, но не имеет собственной User Story в Specification.

7. **Разбить US-C01 на две истории:** участие в занятии и геймификацию --- разные доменные области.

8. **Согласовать поисковый движок:** PRD указывает Meilisearch, Architecture --- Elasticsearch. Зафиксировать выбор в обоих документах.

9. **Добавить измеримые критерии в US-P04:** время загрузки карточки, SEO-требования, поведение при отсутствии данных.

### Приоритет 3 (Улучшения --- можно отложить)

10. **Расширить US для дашбордов:** US-P08 покрывает только прогресс ребёнка; расписание, платежи, управление профилями детей не описаны.

11. **Добавить US для аналитики учителя:** в дашборде учителя (F7 PRD) есть аналитика, но в Specification нет соответствующей US.

12. **Описать accessibility (WCAG 2.1 AA) в конкретных AC:** NFR 3.5 описывает WCAG, но ни одна US не содержит конкретных accessibility-критериев.

13. **Добавить критерии локализации:** указать обработку часовых поясов в streak и расписании (уже упомянуто в PRD F9, но не в AC).

14. **US-P10 (реферальная программа):** отнести к v1.0 (согласно Feature Matrix) или явно включить в MVP, если решено иначе.
